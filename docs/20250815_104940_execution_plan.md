## Phase 1 — Quick Wins and Foundations (1–2 days)
1. Standardize project paths
   - Implement `src/utils/paths.py`
     - `get_project_root()`, `get_data_dir(kind="processed/real"|"processed/unified_real")`, `get_logs_dir()`
   - Wire into (Streamlit first, then Dash):
     - `src/dashboard/streamlit/enhanced_app.py`
     - `src/dashboard/dash/enhanced_app.py`
     - `dashboard_app.py`
     - `src/dashboard/streamlit/data_adapter.py`
   - Acceptance: No hardcoded relative paths; both apps use the same helpers

2. Parquet-first loading and graceful CSV fallback
   - Update loaders to try parquet first, fallback to CSV (single warning)
   - Order: Streamlit → Dash → Dashboard standalone
   - Acceptance: When parquet exists, CSV isn’t used; a single warning logs fallback usage

3. Logging consistency
   - Replace `print` with logging in ETL and scripts (`src/data_extraction/create_unified_real_data.py`, etc.)
   - Ensure `logs/` dir exists prior to any `FileHandler`
   - Make log level/dir env-driven (`LOG_LEVEL`, `LOG_DIR`) with sane defaults
   - Acceptance: No prints in ETL; logs written to `logs/`

4. Oracle client config hardcoding removal
   - In `src/data_extraction/full_data_extractor.py`:
     - Read `ORACLE_CLIENT_LIB` env; only call `cx_Oracle.init_oracle_client` if set
     - Otherwise rely on `LD_LIBRARY_PATH` and skip init
   - Acceptance: Extractor runs with or without explicit client lib path when environment is set

5. Env variable alignment
   - Support both `CHOICE_*` and `ORACLE_*` env names in the extractor; precedence `CHOICE_*` → `ORACLE_*`
   - Emit a single startup warning if falling back
   - Acceptance: Either naming scheme supported; clear logs show which was used

6. Metric standardization (primary = weight)
   - Add dashboard config: `src/dashboard/config.py`
     - `PRIMARY_VOLUME_METRIC = "weight"`
     - `UNITS = {"weight":"lbs", "weight_display":"tons", "units":"count"}`
   - Update visualizations (Streamlit first, then Dash):
     - Donor Performance: bars = total weight; second series = avg weight per donation OR donation count (if count, clearly labeled on secondary axis)
     - Sankey: title/labels “by Total Weight”; enforce consistent unit display
     - Axis titles/tooltips include explicit units (e.g., “Total Weight (tons)”) 
   - Acceptance: All major charts consistently use weight as primary; any count-based metric clearly labeled and secondary

7. Eliminate all data simulations from code (critical)
   - Remove or fully gate all mock/simulated data paths:
     - `src/dashboard/streamlit/data_adapter.py` (remove synthetic demographics; no random generation)
     - `src/dashboard/streamlit/enhanced_app.py` (no inline mock)
     - `src/dashboard/dash/enhanced_app.py` (no inline mock)
   - Any fallback must surface a clear error (and guidance) rather than silently simulate data
   - Acceptance: No random/synthetic data generation; dashboards fail-fast with clear guidance if required data missing

8. Documentation updates
   - Update audit report “Next steps” if needed
   - Add `docs/KPI_STANDARDIZATION_GUIDE.md`
   - Add `docs/DATA_SIMULATION_POLICY.md` (explicit prohibition and acceptable dev practices)
   - Acceptance: Docs aligned with code decisions

## Phase 3 — Codebase cleanup and modularization (3–5 days)
1. Streamlit-first modularization (then Dash replication)
   - Create modules under `src/dashboard/modules/`:
     - `charts.py` (donor_performance, monthly_trends, storage_sunburst, sankey_flow)
     - `metrics.py` (aggregations from unified datasets)
     - `labels.py` (unit conversions + label builders using PRIMARY_VOLUME_METRIC)
     - `loaders.py` (shared parquet-first readers using utils/paths)
     - `ui.py` (shared UI helpers, if needed)
   - Refactor `src/dashboard/streamlit/enhanced_app.py` to import and use these functions
   - Then replicate the same module usage in `src/dashboard/dash/enhanced_app.py`
   - Acceptance: App files are thin; core logic lives in shared modules; no feature regressions

2. Clean up entire `src/` of non-active files
   - Identify and remove or archive files not actively involved in ETL or app generation:
     - Move `src/deprecated/` out of `src/` to `archive/` or `docs/legacy_code/` (outside `src/`)
     - Remove experimental/backup files (e.g., `pages_old_backup`, `app.py.backup`), or move to archive
   - Create `docs/REMOVALS_LOG.md` detailing moved/removed files and rationale
   - Acceptance: `src/` contains only active ETL, shared modules, and app code; deprecated content moved; imports still resolve

3. Validation guardrails
   - On app startup, run a metric consistency check:
     - Scan configured charts for unit-family mismatches (weight vs count) in primary axes
     - Log a warning if detected; render a visible banner in dev mode
   - Acceptance: Detects and warns about accidental metric mismatches

## Execution logistics

- When to assess

  - Per-PR: any change to data loading, KPI logic, core charts, ETL extraction/transform, or logging/error handling

- How feedback is delivered
  - Primary: GitHub PR reviews with inline comments, pass/block decision, severity (Critical/High/Medium/Low), and action items

## Acceptance criteria summary
- No simulated/mocked data anywhere in production code paths
- Streamlit-first development; Dash mirrors after validation
- Primary KPI = Total Gross Weight; labels/units explicit; Sankey/Donor charts consistent
- Data loaders standardized through `utils/paths` + parquet-first
- ETL logs correctly; Oracle client config env-driven
- `src/` contains only active ETL and app code; legacy moved outside `src/`
- Apps modularized: app files thin, reusable modules under `src/dashboard/modules`
- Tests pass locally; CI running; docs updated

## Risks and mitigations
- Removal of files causing unexpected imports to fail
  - Mitigation: staged cleanup with `REMOVALS_LOG.md`; search imports, run apps/tests before merging
- KPI change impacts stakeholder expectations
  - Mitigation: document KPI policy; add clear UI labels and notes
- Streamlit/Dash divergence during replication
  - Mitigation: shared modules for loaders/charts/labels reduce drift
